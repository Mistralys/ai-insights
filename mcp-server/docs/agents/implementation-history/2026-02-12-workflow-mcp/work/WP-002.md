# WP-002: Zod Schemas

## Wave
Wave 1 -- Foundation + Read Tools

## Description
Create Zod schemas that exactly mirror the split-file ledger architecture defined in `personas/ledger/project-ledger-schema.md`. These schemas serve as the single source of truth for runtime validation across all MCP tools. Every field, enum value, and nesting level must match the authoritative schema reference document.

## Requirements

1. **Enums** at `mcp-server/src/schema/enums.ts`
   - `ProjectStatus`: `READY`, `IN_PROGRESS`, `COMPLETE`, `BLOCKED`
   - `WorkPackageStatus`: `READY`, `IN_PROGRESS`, `COMPLETE`, `BLOCKED`
   - `PipelineStatus`: `READY`, `IN_PROGRESS`, `PASS`, `FAIL`
   - `BlockerType`: `dependency`, `decision`, `external`, `technical`
   - `CommentPriority`: `low`, `medium`, `high`

2. **Work Package Detail schema** at `mcp-server/src/schema/work-package.ts`
   - Must validate `ledger/WP-###.json` files exactly as documented in the schema reference
   - Fields: `work_package_id`, `work_package_file`, `status`, `assigned_to`, `dependencies`, `blocked_by` (optional), `acceptance_criteria`, `revision`, `pipelines`
   - Nested schemas: blocker object, acceptance criterion, pipeline (with optional artifacts, metrics, comments)
   - Pipeline comments do NOT have an `agent` field (agent is inferred from pipeline type)
   - The `context` object is only present on `incident` type comments

3. **Root Index schema** at `mcp-server/src/schema/root-index.ts`
   - Must validate `project-ledger.json` exactly as documented
   - Fields: `plan_file`, `date_created`, `last_updated`, `status`, `total_work_packages`, `pending_work_packages`, `work_packages` (array of summary objects), `project_comments`
   - Work package summary: `work_package_id`, `status`, `assigned_to`, `dependencies`, `file`
   - Project comments include `agent` field (unlike pipeline comments)
   - Project comments of type `incident` include optional `context` object with: `os`, `tool`, `work_package` (optional), `resolved`, `workaround` (optional)

4. **Validators** at `mcp-server/src/schema/validators.ts`
   - Export function `isValidStatusTransition(from: WorkPackageStatus, to: WorkPackageStatus): boolean` implementing the transition table from the plan:
     - READY -> IN_PROGRESS (allowed if dependencies met)
     - READY -> BLOCKED
     - IN_PROGRESS -> COMPLETE (allowed if all acceptance criteria met)
     - IN_PROGRESS -> BLOCKED
     - BLOCKED -> IN_PROGRESS
     - COMPLETE -> IN_PROGRESS (triggers revision increment)
     - All other transitions rejected
   - Export function `canStartWorkPackage(wp, allWpSummaries): { allowed: boolean; reason?: string }` that checks whether all dependency WPs have status COMPLETE
   - Export function `canCompleteWorkPackage(wp): { allowed: boolean; unmet?: string[] }` that checks all `acceptance_criteria` have `met: true`

## Technical Constraints
- Use `zod` for all schema definitions -- no hand-rolled validation.
- Schemas must parse real existing ledger files without errors. The Developer Agent should test against any existing `project-ledger.json` and `ledger/WP-###.json` files in the repo.
- Schemas should be strict (no extra keys allowed) via `.strict()` where appropriate, but lenient on optional fields that may not be present in older ledger files.

## Acceptance Criteria
- [ ] All enum values match the schema reference document exactly.
- [ ] Root index schema successfully parses a valid `project-ledger.json` fixture.
- [ ] Work package detail schema successfully parses a valid `ledger/WP-###.json` fixture.
- [ ] Schemas reject malformed input (missing required fields, invalid enum values, wrong types).
- [ ] `isValidStatusTransition` accepts all 6 legal transitions and rejects illegal ones.
- [ ] `canStartWorkPackage` returns `allowed: false` when any dependency WP is not COMPLETE.
- [ ] `canCompleteWorkPackage` returns `allowed: false` with list of unmet criteria when any criterion has `met: false`.
- [ ] TypeScript compiles without errors.
