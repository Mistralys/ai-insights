# WP-005: Core Write Tools

## Wave
Wave 2 -- Core Write Operations

## Description
Implement the write-side MCP tools for project initialization and work package management. These tools enforce status transition rules, dependency validation, and dual-file atomicity -- eliminating the desync bugs that motivated this project.

## Requirements

1. **initialize_project** in `mcp-server/src/tools/project-lifecycle.ts`
   - Input: `{ project_path: string, plan_file: string }`
   - Behavior:
     - Validates that the plan folder exists at `project_path`
     - Rejects if `project-ledger.json` already exists (prevents accidental overwrite)
     - Creates the `ledger/` subdirectory if it does not exist
     - Creates `project-ledger.json` with: `plan_file`, `date_created` (now), `last_updated` (now), `status: "READY"`, `total_work_packages: 0`, `pending_work_packages: 0`, empty `work_packages` array, empty `project_comments` array
     - Returns the created root index
   - Error cases: plan folder does not exist, ledger already exists

2. **create_work_package** in `mcp-server/src/tools/work-package.ts`
   - Input: `{ project_path: string, assigned_to: string, dependencies: string[], acceptance_criteria: string[], work_package_file: string }`
   - Behavior:
     - Reads root index to determine next WP ID (length of work_packages array + 1, formatted as `WP-###`)
     - Creates `ledger/WP-###.json` with: auto-generated `work_package_id`, provided `work_package_file`, `status: "READY"`, provided `assigned_to`, provided `dependencies`, `acceptance_criteria` array (each criterion with `met: false`), `revision: 1`, empty `pipelines` array
     - Appends a summary entry to the root index `work_packages` array: `work_package_id`, `status: "READY"`, `assigned_to`, `dependencies`, `file: "ledger/WP-###.json"`
     - Increments `total_work_packages` and `pending_work_packages` in root index
     - Sets project status to `IN_PROGRESS` if it was `READY`
     - Updates `last_updated`
     - All writes via `LedgerStore` within a single lock
   - Error cases: root index missing, invalid dependency WP IDs (must reference existing WPs)

3. **claim_work_package** in `mcp-server/src/tools/work-package.ts`
   - Input: `{ project_path: string, work_package_id: string, agent: string }`
   - Behavior:
     - Uses `canStartWorkPackage` validator to check all dependencies are COMPLETE
     - Transitions WP status from READY to IN_PROGRESS via `isValidStatusTransition`
     - Sets `assigned_to` to the claiming agent
     - Updates both WP detail file and root index summary atomically
     - Updates `last_updated`
   - Error cases: WP not in READY status, dependencies not met, invalid WP ID

4. **update_work_package_status** in `mcp-server/src/tools/work-package.ts`
   - Input: `{ project_path: string, work_package_id: string, status: string, blocked_by?: { type: string, description: string, blocking_work_package?: string } }`
   - Behavior:
     - Validates the transition via `isValidStatusTransition`
     - For transition to COMPLETE: uses `canCompleteWorkPackage` to verify all acceptance criteria are met
     - For transition to BLOCKED: requires `blocked_by` object
     - For transition from COMPLETE to IN_PROGRESS: increments `revision`
     - For transition from BLOCKED to IN_PROGRESS: clears `blocked_by`
     - Updates `pending_work_packages` counter in root index when completing (decrement) or un-completing (increment)
     - All updates atomic via `updateWorkPackageWithSync`
     - Updates `last_updated`
   - Error cases: invalid transition, unmet acceptance criteria for COMPLETE, missing blocked_by for BLOCKED

## Technical Constraints
- All write operations must use `LedgerStore.updateWorkPackageWithSync` or acquire a file lock to ensure dual-file consistency.
- Status transition validation must happen BEFORE any file writes.
- Error messages must be actionable -- tell the agent what state the WP is in, what transition was attempted, and what is required.

## Acceptance Criteria
- [ ] `initialize_project` creates a valid root index and ledger directory.
- [ ] `initialize_project` rejects when ledger already exists.
- [ ] `create_work_package` creates both the WP detail file and root index summary entry atomically.
- [ ] `create_work_package` auto-generates sequential WP IDs.
- [ ] `claim_work_package` transitions READY to IN_PROGRESS only when dependencies are met.
- [ ] `claim_work_package` rejects when dependencies are not COMPLETE.
- [ ] `update_work_package_status` enforces all 6 legal status transitions.
- [ ] `update_work_package_status` rejects illegal transitions with descriptive error.
- [ ] `update_work_package_status` requires `blocked_by` for BLOCKED transitions.
- [ ] `update_work_package_status` increments revision on COMPLETE -> IN_PROGRESS.
- [ ] `pending_work_packages` counter stays accurate across all status changes.
- [ ] TypeScript compiles without errors.
