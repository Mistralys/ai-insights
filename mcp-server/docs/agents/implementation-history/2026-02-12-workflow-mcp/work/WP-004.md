# WP-004: Read Tools

## Wave
Wave 1 -- Foundation + Read Tools

## Description
Register the first MCP tools on the server: `get_project_status`, `get_work_package`, and `list_work_packages`. These are read-only tools that allow agents to query ledger state via MCP with full schema validation. The `get_project_status` tool includes self-healing logic that recomputes counters from actual work package data.

## Requirements

1. **Tool registration pattern**: Establish the pattern for registering tools on the `McpServer` instance from `index.ts`. Each tool module exports a `register(server: McpServer)` function that calls `server.tool(name, schema, handler)`.

2. **get_project_status** in `mcp-server/src/tools/project-lifecycle.ts`
   - Input: `{ project_path: string }` (Zod-validated)
   - Behavior:
     - Reads the root index via `LedgerStore.readRootIndex()`
     - Recomputes `pending_work_packages` by counting WP summaries where status is not COMPLETE
     - Recomputes `total_work_packages` from actual array length
     - If recomputed values differ from stored values, updates the root index (self-healing)
     - Returns the full root index object
   - Error cases: project path does not exist, root index missing, malformed JSON

3. **get_work_package** in `mcp-server/src/tools/work-package.ts`
   - Input: `{ project_path: string, work_package_id: string }` (Zod-validated, WP ID must match `WP-###` format)
   - Behavior:
     - Reads the WP detail file via `LedgerStore.readWorkPackage()`
     - Returns the full work package detail object
   - Error cases: WP file does not exist, malformed JSON, invalid WP ID format

4. **list_work_packages** in `mcp-server/src/tools/work-package.ts`
   - Input: `{ project_path: string, status?: string, assigned_to?: string }` (Zod-validated)
   - Behavior:
     - Reads root index to get work package summary array
     - Filters by status and/or assigned_to if provided
     - Returns filtered array of WP summary objects
   - Error cases: root index missing

5. **Wire tools into index.ts**: Import and call the register functions in the entry point so tools are available when the server starts.

6. **Test fixtures** at `mcp-server/tests/fixtures/`
   - Create at least one valid `project-ledger.json` fixture and one valid `ledger/WP-001.json` fixture matching the schema
   - These fixtures are used by unit/integration tests for all subsequent WPs as well

## Technical Constraints
- Tool handlers must return MCP-compatible responses (content array with text type).
- All tool inputs validated with Zod before processing.
- Errors returned as MCP tool errors with `isError: true` and actionable messages.
- Read tools do not acquire file locks (only write paths need locking).

## Acceptance Criteria
- [ ] `get_project_status` returns a valid root index object for a test project.
- [ ] `get_project_status` self-heals incorrect `pending_work_packages` and `total_work_packages` counts.
- [ ] `get_work_package` returns the full WP detail for a valid WP ID.
- [ ] `get_work_package` returns an MCP error for a non-existent WP.
- [ ] `list_work_packages` returns all WPs when no filters provided.
- [ ] `list_work_packages` correctly filters by status and assigned_to.
- [ ] All three tools are accessible when the MCP server starts.
- [ ] Test fixtures are valid against the Zod schemas.
- [ ] TypeScript compiles without errors.
