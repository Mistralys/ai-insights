# WP-003: Storage Layer

## Wave
Wave 1 -- Foundation + Read Tools

## Description
Implement the storage layer that all MCP tools will use for reading and writing ledger files. This layer provides file locking, atomic writes (write-to-temp-then-rename), and a unified `LedgerStore` abstraction that enforces dual-file synchronization between the root index and individual work package files.

## Requirements

1. **FileLock** at `mcp-server/src/storage/file-lock.ts`
   - Wrap `proper-lockfile` to lock/unlock a project's ledger directory
   - Lock file location: `{project_path}/.ledger.lock`
   - Configuration: 10-second stale timeout, 5 retries, suitable retry interval
   - Export an async `withLock<T>(projectPath: string, fn: () => Promise<T>): Promise<T>` helper that acquires the lock, runs the callback, and releases in a `finally` block
   - Must handle the case where the lock file does not yet exist (create it)

2. **AtomicWriter** at `mcp-server/src/storage/atomic-writer.ts`
   - Export async function `atomicWriteJson(filePath: string, data: unknown): Promise<void>`
   - Writes JSON to `{filePath}.tmp.{pid}` then uses `fs.rename` for POSIX-atomic replacement
   - Pretty-prints JSON with 2-space indentation and trailing newline
   - Cleans up temp file on error

3. **LedgerStore** at `mcp-server/src/storage/ledger-store.ts`
   - Central abstraction for all ledger file I/O
   - Constructor takes `projectPath: string`
   - **Read methods** (all validate with Zod schemas from WP-002):
     - `readRootIndex(): Promise<RootIndex>` -- reads and validates `project-ledger.json`
     - `readWorkPackage(wpId: string): Promise<WorkPackageDetail>` -- reads and validates `ledger/WP-###.json`
   - **Write methods** (all use AtomicWriter and FileLock):
     - `writeRootIndex(data: RootIndex): Promise<void>` -- validates then writes root index
     - `writeWorkPackage(wpId: string, data: WorkPackageDetail): Promise<void>` -- validates then writes WP detail file
     - `updateWorkPackageWithSync(wpId: string, updater: (wp: WorkPackageDetail, root: RootIndex) => { wp: WorkPackageDetail; root: RootIndex }): Promise<void>` -- the critical dual-file update method. Reads both files, applies the updater function, writes both atomically within a single lock. This is the method that prevents the desync bugs described in the plan context.
   - **Path helpers** (internal):
     - `rootIndexPath()` returns `{projectPath}/project-ledger.json`
     - `wpDetailPath(wpId)` returns `{projectPath}/ledger/WP-###.json`
     - `ledgerDirPath()` returns `{projectPath}/ledger/`
   - **Existence checks**:
     - `rootIndexExists(): Promise<boolean>`
     - `wpDetailExists(wpId: string): Promise<boolean>`
     - `ledgerDirExists(): Promise<boolean>`

4. **Error handling**:
   - Throw descriptive errors when files are not found, JSON is malformed, or Zod validation fails
   - Include the file path and the Zod error details in error messages
   - Lock acquisition failures should include retry count and timeout info

## Technical Constraints
- All file operations use Node.js `fs/promises`.
- `fs.rename` is used for atomic replacement (POSIX semantics).
- The `withLock` + `updateWorkPackageWithSync` combination must guarantee that no partial writes occur -- both files are written or neither is.
- Do not cache file contents in memory between tool calls; always read fresh from disk (MCP tools are stateless between calls).

## Acceptance Criteria
- [ ] `withLock` acquires and releases the lock correctly (test with concurrent access simulation).
- [ ] `atomicWriteJson` produces correctly formatted JSON and cleans up temp files.
- [ ] `readRootIndex` returns validated data matching the Zod schema.
- [ ] `readWorkPackage` returns validated data matching the Zod schema.
- [ ] `updateWorkPackageWithSync` updates both files atomically within a single lock.
- [ ] Descriptive errors are thrown for missing files, malformed JSON, and validation failures.
- [ ] TypeScript compiles without errors.
