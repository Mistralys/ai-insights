# WP-007: Workflow Intelligence Tools

## Wave
Wave 4 -- Workflow Intelligence

## Description
Implement the two workflow intelligence tools that help agents decide what to do next and compute correct handoff status lines. These tools encode the agent workflow logic that agents currently miscalculate, causing incorrect STATUS lines at the end of their responses.

## Requirements

1. **get_next_action** in `mcp-server/src/tools/workflow.ts`
   - Input: `{ project_path: string, agent_role: string }`
   - Behavior:
     - Reads the root index to assess overall project state
     - Based on the `agent_role`, examines work package statuses and pipeline states to recommend the next action
     - Agent role mapping (based on the 7-stage workflow):
       - `"Project Manager"`: Look for READY project with no WPs yet -> create WPs. Look for all WPs COMPLETE -> signal synthesis.
       - `"Developer"`: Look for READY or unblocked WPs with no implementation pipeline -> claim and implement. Look for FAIL pipelines needing rework.
       - `"QA"`: Look for WPs with PASS implementation pipeline but no QA pipeline -> run QA.
       - `"Reviewer"`: Look for WPs with PASS QA pipeline but no code-review pipeline -> run review.
       - `"Documentation"`: Look for WPs with PASS code-review pipeline but no docs pipeline -> write docs.
       - `"Synthesis"`: Check if all WPs are COMPLETE -> generate synthesis report.
     - Returns a structured recommendation: `{ action: string, work_package_id?: string, reason: string }`
     - If no action available, returns `{ action: "WAIT", reason: "..." }`
   - Error cases: root index missing, unknown agent role

2. **get_handoff_status** in `mcp-server/src/tools/workflow.ts`
   - Input: `{ project_path: string, current_agent: string }`
   - Behavior:
     - Reads the root index and examines all WP statuses and pipelines
     - Computes the correct `AGENT:` and `STATUS:` handoff block that the current agent should emit at the end of their response
     - Logic:
       - If any WP is BLOCKED -> `STATUS: BLOCKED` with description
       - If current agent's work is done and next stage is ready -> `STATUS: READY_FOR_{NEXT_AGENT}` (e.g., `READY_FOR_QA`, `READY_FOR_REVIEW`)
       - If all WPs are COMPLETE -> `STATUS: READY_FOR_SYNTHESIS` or `STATUS: COMPLETE`
       - If work remains for the current agent -> `STATUS: IN_PROGRESS`
     - Returns: `{ agent: string, status: string, details?: string }`
   - Error cases: root index missing, unknown agent

3. **Wire both tools into index.ts** via the register pattern established in WP-004.

## Technical Constraints
- These tools are read-only -- they examine state but do not modify files.
- The workflow logic must align with the 7-stage agent workflow documented in the persona system (Planner -> PM -> Developer -> QA -> Reviewer -> Docs -> Synthesis).
- Recommendations must be actionable and specific (include WP IDs, not just generic advice).
- The handoff status computation must handle edge cases: partially complete projects, multiple WPs in different states, blocked WPs coexisting with ready WPs.

## Acceptance Criteria
- [ ] `get_next_action` returns correct recommendations for each agent role given a test project state.
- [ ] `get_next_action` returns `WAIT` when no actionable work exists for the given role.
- [ ] `get_next_action` correctly identifies WPs needing rework (FAIL pipelines).
- [ ] `get_handoff_status` computes correct STATUS line for a project with all WPs COMPLETE.
- [ ] `get_handoff_status` computes correct STATUS line for a project with mixed WP states.
- [ ] `get_handoff_status` returns BLOCKED status when any WP is blocked.
- [ ] Both tools handle edge cases (empty project, single WP, all blocked).
- [ ] TypeScript compiles without errors.
